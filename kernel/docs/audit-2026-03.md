## 1. BUGS

### B1. `vmfree()` prints wrong value in panic message
**File:** `kernel/mman.c:218`
```c
panic("Attempted to free unknown vma %h", a);
```
`a` is `NULL` at this point (the search failed to find a match). The format specifier prints the pointer `a`, which is always `0x00000000`. Should print `ptr` instead.

**Fix:** Change to `panic("Attempted to free unknown vma %h", ptr);`

---

### B2. `vm_unmap()` page table sweep may access wrong PD entry on boundary
**File:** `kernel/mman.c:174`
```c
pde_t last = VA_PD_IDX(vm->start + vm->size);
for (pde_t p = start; p <= last; ++p) {
```
If `vm->start + vm->size` is exactly page-directory aligned (e.g., a multiple of 4MB), `VA_PD_IDX` returns the index of the *next* PD entry, not the last one belonging to this VMA. The `<=` loop then checks one PDE too many, potentially freeing a page table that belongs to another mapping.

**Fix:** Use `VA_PD_IDX(vm->start + vm->size - 1)` or change to `<`.

---

### B3. `kprintf` format parsing breaks on adjacent format specifiers
**File:** `kernel/kprintf.c:113-176`

After processing a format specifier (e.g. `%i`), the code does `i += 2` to skip past it. Then, after the switch statement, `terminal_putchar(fmt[i])` prints the character at position `i` as literal text. This means every format specifier unconditionally consumes the *next* character as literal output.

This is correct when format specifiers are followed by normal text (e.g. `"%i items"` works fine -- the space gets printed). But it breaks when format specifiers are adjacent:

**Example:** `kprintf("%i%i", 1, 2)` outputs `1%i` instead of `12`. The second `%` is consumed as a literal character, and then `i` is printed literally on the next loop iteration.

---

### B4. `try_get_num()` uses uninitialized `places` array (undefined behavior)
**File:** `kernel/kprintf.c:62-68`
```c
uint32_t places[11];
size_t idx = 0;
while (is_num(str[idx])) {
    for (size_t i = 0; i < 11; ++i)
        places[i] *= 10;
    places[idx] = str[idx] - '0';
    idx++;
}
```
`places` is a stack array that is never initialized to zero. On the first iteration, `places[i] *= 10` reads uninitialized memory, which is undefined behavior per the C standard. In practice the algorithm happens to produce correct results because the garbage-multiplied entries are always overwritten before being summed, but the compiler is free to exploit UB and break this code.

Additionally, there is no bounds check on `idx`: a format string with 11+ consecutive digits (e.g. `%12345678901s`) overflows the `places[11]` array.

**Fix:** Initialize: `uint32_t places[11] = {0};` and add `if (idx >= 11) break;`

---

### B5. `places()` has signed overflow for large unsigned values
**File:** `kernel/kprintf.c:14-20`
```c
static int places(int n) {
    int places = 1;
    while (n >= 10) { ... }
    return places;
}
```
Called by `pru32()` with a `uint32_t`, but `places()` takes `int`. For values > `INT_MAX` (~2.1 billion), the implicit conversion is implementation-defined and typically produces a negative value. The `while (n >= 10)` check then fails immediately, returning `places = 1`, causing the printed number to be truncated.

### B6. `pf_have_frames()` dereferences NULL on empty freelist
**File:** `kernel/pfa.c:178-182`
```c
int pf_have_frames(size_t n) {
    struct page_frame *page = page_freelist;
    for (size_t i = 0; i < n; ++i)
        if (!(page = page->next))
            return 0;
    return 1;
}
```
If `page_freelist` is `NULL` and `n > 0`, the first iteration dereferences `NULL` (`page->next` where `page` is `NULL`).

**Fix:** Check `if (!page || !(page = page->next))`.

### B7. `do_syscall()` has no bounds check on syscall number
**File:** `kernel/syscalls.c:59`
```c
struct syscall sc = syscalls[regs.eax];
```
`regs.eax` comes directly from userspace and is used as an array index with no bounds checking. A malicious or buggy user program can cause an out-of-bounds read. The `syscalls` array only has entries up to index 47.

**Fix:** Add `if (regs.eax >= sizeof(syscalls)/sizeof(syscalls[0])) { /* terminate */ }`

---

### B8. `do_syscall()` never returns result to userspace
**File:** `kernel/syscalls.c:82-83`
```c
// FIXME: pass ret in eax to task
(void)ret;
```
The syscall return value is computed but never stored back into the interrupted task's register state. Every syscall returns an undefined value to the caller. This makes the syscall interface non-functional for any syscall that needs to return a value.

**Fix:** Write `ret` back to `regs.eax` (requires passing regs by pointer rather than by value, or accessing the stack frame directly).

### B9. `PROT_USR` value collides with `PROT_READ | PROT_WRITE`
**File:** `kernel/include/mman.h:139-143`
```c
#define PROT_READ  0x1
#define PROT_WRITE 0x2
#define PROT_NONE  0x0
#define PROT_USR   0x3  // <-- 0x3 == PROT_READ | PROT_WRITE
```
`PROT_USR` has value `0x3`, which is the same as `PROT_READ | PROT_WRITE`. In `mprotect()`, the check `if (prot & PROT_USR)` will always be true when both READ and WRITE are set, even if user mode wasn't intended.

**Fix:** Use `0x4` for `PROT_USR` so it occupies its own bit.

The `%i` format specifier has no corresponding argument. This reads a garbage value from the stack.

### B11. `cmd_sense()` incorrectly waits for IRQ
**File:** `kernel/drivers/floppy.c:1176`
```c
ret = wait_irq();
```
`CMD_SENSE_DRIVE_STATUS` does not generate an IRQ (it's a non-interrupt command per the 8272A spec). This `wait_irq()` call will always time out, causing the function to return an error unnecessarily.

### B12. `find_dirent()` name matching is prefix-based, not exact
**File:** `kernel/fs/ext2/ext2.c:593`
```c
if (strncmp(dirent->name, name, strlen(name)) == 0) {
```
This matches any directory entry whose name *starts* with `name`. For example, searching for `"foo"` would also match `"foobar"`. Should also check that `dirent->name_length_lsb == strlen(name)`.

### B13. `read_i_block()` ignores `read_block()` return value
**File:** `kernel/fs/ext2/ext2.c:284,289,295`
```c
int ret = read_block(fs, i->singly_linked, block);
return read_indirect(fs, i, idx, data, (blkaddr_t *)block, 0);
```
`ret` from `read_block` is never checked on the singly, doubly, and triply linked paths. If the read fails, `read_indirect` operates on garbage data.

### B14. `task_kill()` doesn't restore interrupts on early return
**File:** `kernel/sched.c:257-268`
```c
int task_kill(tid_t id) {
    if (id < 1)
        return -1;
    cli_push();
    struct task *to_kill = find_task(id);
    if (!to_kill)
        return -1;  // <-- cli_pop() never called!
```
If `find_task()` returns NULL, the function returns -1 without calling `cli_pop()`, leaving interrupts permanently disabled.

### B15. IDT/GDT limit fields are off by one
**File:** `kernel/idt.c:59`
```c
.limit = IDT_ENTRIES * sizeof(idt_entries[0]),
```
The `limit` field in the IDTR/GDTR is supposed to be `(size - 1)`, i.e. the offset of the last valid byte. The current code sets it one byte too large. Same issue in `kernel/x86.c:63` for the GDT.

**Fix:** `.limit = IDT_ENTRIES * sizeof(idt_entries[0]) - 1,`

### B16. `terminal_putchar()` sends backspace to serial twice
**File:** `kernel/terminal.c:137-158`

The backspace handler (lines 137-148) calls `write_serial(c)` at line 147. Then execution falls through to the unconditional `write_serial(c)` at line 158. This sends the backspace character to the serial port twice.

### B17. `serial_debug.c` IRQ handler unconditionally panics
**File:** `kernel/serial_debug.c:98-100`
```c
static void do_serial(struct irq_regs regs) {
    panic("serial");
}
```
If a serial interrupt fires (e.g., data received on COM2), the kernel panics. The handler is registered via `attach_irq()` during `serial_setup()`, so this is a live IRQ that will crash the system if triggered.

### B18. `attach_irq()` bounds check is off by one
**File:** `kernel/idt.c:318`
```c
if (irq < 0 || irq > num_irqs)
```
Should be `irq >= num_irqs` since valid indices are `0` to `num_irqs - 1`. As written, `irq == num_irqs` passes the check and causes an out-of-bounds access to `irq_handlers[num_irqs]`.

### B19. `mprotect()` corrupts PDE flags for other mappings
**File:** `kernel/mman.c:258-259`
```c
*pde = *pde & ~0xFFF;
*pde |= flags;
```
This replaces *all* PDE flags with the new protection flags. The PDE controls access to an entire 4MB region (1024 page table entries). If other pages sharing the same page table have different permissions, setting restrictive PDE flags (e.g., removing PTE_PRESENT or PTE_WRITABLE) could deny access to those unrelated pages.

### B20. `reaper` variable shadowing corrupts outer safe-iteration loop
**File:** `kernel/sched.c:45,60`
```c
// Line 45:
v_ilist *pos, *temp;
v_ilist_for_each_safe(pos, temp, &stop_queue) {
    ...
    // Line 60:
    v_ilist *pos, *temp;
    v_ilist_for_each_safe(pos, temp, &t->waiters) {
```
The inner `pos` and `temp` at line 60 shadow the outer loop's variables at line 45. After the inner loop finishes, the outer loop's `temp` pointer (which `v_ilist_for_each_safe` relies on for safe iteration) has been clobbered. This can cause the outer loop to skip elements, revisit elements, or crash.

## 2. RISKS

### R1. `vmalloc()` calls `vm_map()` outside the critical section
**File:** `kernel/mman.c:198-203`
```c
cli_push();
struct vma *vma = find_space(pages);
cli_pop();
...
vm_map(vma);
```
`vm_map()` modifies page tables and calls `pf_alloc()` outside the critical section. If a timer interrupt fires during `vm_map()` and triggers a context switch, another task could concurrently modify page tables or the page frame allocator, causing corruption.

---

### R2. Page frame allocator is not thread-safe
**File:** `kernel/pfa.c:185-201`

`pf_alloc()` and `pf_free()` manipulate a shared freelist with no locking or interrupt disabling. With preemptive multitasking, concurrent access from different tasks (or from interrupt context) can corrupt the freelist.

---

### R3. `sleep()` panics on timer wraparound
**File:** `kernel/timer.c:46-53`
```c
current->sleep_till = system_uptime_ms + ms;
...
if (!current->sleep_till)
    panic("!current->sleep_till");
current->sleep_till = 0;
```
If `system_uptime_ms + ms` overflows to exactly 0 (after ~49.7 days uptime), `sleep_till` is set to 0. The task will wake immediately, and the assertion at line 51 fires, crashing the kernel.

---

### R4. `system_uptime_ms` is not `volatile`
**File:** `kernel/timer.c:28` and `kernel/include/timer.h:7`

This variable is modified in the timer interrupt handler (`do_timer()`) but read from normal code (e.g., `find_next_runnable()`, `sleep()`, `get_uptime()`). Without `volatile`, the compiler may cache the value in a register and never re-read it from memory, causing infinite loops in `sleep()` or missed wakeups.

---

### R5. `floppy_interrupts` decrement is not atomic
**File:** `kernel/drivers/floppy.c:366`
```c
--floppy_interrupts;
```
`floppy_interrupts` is incremented in the ISR and decremented in `wait_irq()`. The decrement is not atomic and not protected by `cli()`. If an interrupt fires between the load and store of the decrement, an interrupt count is lost.

---

### R6. `motor_ms` is `uint16_t` but compared with `<= 0`
**File:** `kernel/drivers/floppy.c:539-541`
```c
d->motor_ms -= fdc_timeout_granularity;
if (d->motor_ms <= 0) {
```
`motor_ms` is `uint16_t` (unsigned). If `fdc_timeout_granularity` (250) is greater than the remaining `motor_ms`, the subtraction wraps to a large positive value and the `<= 0` check never triggers. The motor never turns off.

---

### R7. `memcpy` and `memset` signatures don't match standard C
**File:** `kernel/include/utils.h:16-17`
```c
void *memcpy(unsigned char *dst, unsigned char *src, size_t bytes);
void *memset(unsigned char *dst, unsigned char c, size_t bytes);
```
Standard signatures use `void *` and `const void *`. GCC may generate implicit calls to `memcpy`/`memset` (e.g., for struct copies or zeroing) using the standard calling convention. Mismatched signatures could cause subtle bugs with link-time optimization.

---

### R8. Keyboard shift handling is incorrect for multi-key scenarios
**File:** `kernel/keyboard.c:180-181`
```c
if (lowerSeven == 0x36 || scancode == 0x2A || scancode == 0xAA) {
    g_shifted = !g_shifted;
```
This toggles shift state on every shift-related scancode. For right shift (0x36), both keydown (0x36) and keyup (0xB6) have `lowerSeven == 0x36`, so pressing and releasing right shift toggles twice (net zero -- correct). But if both shift keys are pressed simultaneously, the state becomes inverted: press left shift (shifted=1), press right shift (shifted=0, unexpected unshift).

---

### R9. `DMA_BUF_SIZE` macro missing parentheses
**File:** `kernel/drivers/floppy.c:354`
```c
#define DMA_BUF_SIZE 2 * 18 * 512
```
If used in an expression like `sizeof(x) / DMA_BUF_SIZE`, operator precedence would give wrong results. Should be `(2 * 18 * 512)`.

---

### R10. `kmalloc` has no sub-page allocator
**File:** `kernel/mman.c:223-227`
```c
void *kmalloc(size_t bytes) {
    if (bytes <= PAGE_SIZE)
        return pf_alloc();
    return vmalloc(bytes);
}
```
All allocations <= 4096 bytes consume an entire physical page. A 32-byte `struct task` allocation wastes 4064 bytes. There's also a discontinuity: `kmalloc(4096)` returns one page, but `kmalloc(4097)` goes through `vmalloc` which adds overhead pages.

### R11. `console.c` globals are shared across tasks without synchronization
**File:** `kernel/console.c:49,188`

Multiple tasks spawned from the console menu share `s_fgcolor` and `spot_idx`, which are modified without any locking.

---

### R12. `wait_tid()` manipulates lists with interrupts enabled
**File:** `kernel/sched.c:270-283`
```c
int wait_tid(tid_t task_id) {
    ...
    struct task *waitee = find_task(task_id);
    ...
    current->waiting = 1;
    v_ilist_prepend(&current->linkage, &waitee->waiters);
    cli_push();
    sched();
    cli_pop();
```
`find_task()` and the list manipulation happen with interrupts enabled. A timer interrupt could fire between finding the task and adding to its waiters list, causing the waitee to be reaped before we've registered as a waiter, leading to use-after-free.

### R13. `sched()` calls `v_ilist_remove()` on idle_task which may not be in any list
**File:** `kernel/sched.c:346-347`
```c
if (!next)
    next = idle_task;
v_ilist_remove(&next->linkage);
```
`idle_task` is removed from `runqueue` in `sched_init()` and is never re-added. When `find_next_runnable()` returns NULL, calling `v_ilist_remove` on a detached node could corrupt list pointers (depends on `v_ilist_remove` implementation).

### R14. `stage0_init` parameter layout depends on MBR stack layout
**File:** `kernel/stage0.c:67`
```c
extern void _stage0_init(uint16_t mem_kb, uint16_t pad0, uint32_t pad1, uint32_t pad2) {
```
The `pad` parameters exist to account for the MBR pushing values onto the stack before calling this function. If the bootloader's stack layout changes, these pads silently become wrong.

### R15. No TLB invalidation for individual pages
**File:** `kernel/include/mman.h:120-128`

The kernel uses `flush_cr3()` (full TLB flush) everywhere. Between modifying a PTE and calling `flush_cr3()`, stale TLB entries could be used by interrupt handlers. Single-page `invlpg` (available on 486+) would be both safer and more efficient.

### R16. `dyn_array.h` uses `malloc`/`realloc`/`free` unavailable in kernel
**File:** `kernel/include/lib/dyn_array.h:14-15`
```c
#include <stdlib.h>
#include <string.h>
```
This header is in the kernel include tree but uses standard library functions. If any kernel code instantiates a `dyn_array` type, it will fail to link.

### R17. `SYSCALL0` macro references non-existent operand
**File:** `kernel/include/syscalls.h:15-24`

### R18. Floppy timeout task can run forever
**File:** `kernel/drivers/floppy.c:536-550`
```c
int floppy_timeout(void *ctx) {
    struct floppy_drive *d = ctx;
    while (1) {
        sleep(fdc_timeout_granularity);
        if (d->motor_state == fd_mot_timeout) {
            d->motor_ms -= fdc_timeout_granularity;
            ...
```
If `motor_set(d, 1)` is called (setting `motor_state` to `fd_mot_on`) and no subsequent `motor_set(d, 0)` ever occurs, the timeout task loops forever: it sleeps, checks if `motor_state == fd_mot_timeout`, finds it's not, and loops again. The task never exits, wasting a task slot and stack permanently.

### R19. `get_physical_address()` assumes all page tables are in PFA region
**File:** `kernel/mman.c:42`

This adds `PFA_VIRT_OFFSET` to the physical address stored in the PDE. This works as long as all page tables live in the PFA-mapped region (`0xD0000000+`). If page tables are ever allocated outside that region (e.g., from the stage0 identity mapping), this produces an incorrect virtual address.

## 3. IMPROVEMENTS

### I1. `places()` function in kprintf should take `uint32_t`
**File:** `kernel/kprintf.c:14`

Since it's only called from `pru32()` with unsigned values, changing the parameter type to `uint32_t` eliminates the signed overflow issue described in B5.

---

### I2. `memcpy`/`memset` should use standard signatures
**File:** `kernel/utils.c:14,20`

Use `void *dst, const void *src` for `memcpy` and `void *dst, int c` for `memset` to match the C standard and avoid implicit conversions.

---

### I3. Terminal typo: `TERM_HEIGH` should be `TERM_HEIGHT`
**File:** `kernel/terminal.c:13`

Minor typo used consistently throughout the file.

---

### I4. Consider using `invlpg` instead of full CR3 flush
**File:** `kernel/include/mman.h:120`

The 486 supports `invlpg` for single-page TLB invalidation. Using it in `vm_map()`/`vm_unmap()`/`mprotect()` would be significantly cheaper than flushing the entire TLB.

**File:** `kernel/serial_debug.c:199`

**File:** `kernel/drivers/floppy.c:58`

## 4. Cross-cutting Concerns

### Interrupt Safety
The most pervasive issue is inconsistent interrupt protection. The page frame allocator (`pf_alloc`/`pf_free`), scheduler functions (`wait_tid`, `task_kill`), and the floppy driver all manipulate shared state without consistently disabling interrupts. With preemptive multitasking at 250Hz, these are ticking time bombs for hard-to-reproduce corruption bugs.

### Memory Management
The kernel lacks a slab allocator or any sub-page allocator. All `kmalloc` calls for <= PAGE_SIZE waste the remainder of the page. A 32-byte `struct task` allocation burns an entire 4KB page. The VM allocator's arena is only 8 pages (32KB), which limits the total number of tracked VMAs.

### Error Handling
Many functions panic on errors that could be handled gracefully (e.g., `find_space()` panics instead of returning NULL, `mprotect` returns -1 but callers often `assert(!mprotect(...))` anyway). This makes the kernel fragile under memory pressure.

### Userspace Isolation
The current userspace task implementation shares the kernel's address space and page tables. The `mprotect()` hack to mark individual pages as user-accessible is a temporary solution that doesn't provide real isolation. Any user task can read/write all kernel memory that hasn't been specifically marked as supervisor-only.

### kprintf Reliability
The format string parser has multiple issues: adjacent format specifiers are broken (B3), the number parser has UB (B4), and large unsigned integers print incorrectly (B5). Since `kprintf` is the primary debugging tool, these issues make diagnosing other bugs harder.
